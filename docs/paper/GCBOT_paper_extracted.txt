GCBOT: A Gamified Autonomous Garbage Collection Robot utilizing Hybrid Edge-AI and Physical Telemetry

First A S Kavin,a  Second M Mohammed Fayaz,a  Third S Deepak Kumar,a  

And J Prabavadhi.b*

Abstract-- The management of solid waste remains a critical global challenge that requires innovative solutions that encourage public participation. This study presents the Gamified Cleaning Bot (GCBOT), a remotely controlled, multilevel garbage collection robot designed to address waste management through gamification. The system integrates a custom-trained YOLOv10 object detection model for real-time visual classification with an HX711 load cell for synchronized weight measurements. A unique hybrid scoring algorithm dynamically calculates points using the formula Score = Base × (1 + Weight/100), which balances the environmental impact classes with physical mass. The robot architecture utilizes a low-latency edge-to-fog TCP communication pipeline, bridging an Arduino-based hardware controller, an onboard Raspberry Pi for video streaming, and a high-performance host executing deep learning inference and a Flask web interface. By eliminating command lag through TCP optimizations, the GCBOT achieves near-real-time mobile operability, demonstrating a robust foundation for gamified ecological initiatives.

Index Terms-- Gamification, garbage collection robot, YOLOv10, object detection, waste classification, hybrid scoring, load cell, edge computing, TCP socket communication, Flask, MJPEG streaming, robotic arm, Raspberry Pi, Arduino.

The integration of gamification into non-recreational domains is rapidly transforming how societies approach tedious, labor-intensive tasks. By applying game-design elements—such as point scoring, competition, and real-time feedback—to real-world processes, gamification effectively leverages human psychology to drive engagement and motivation. In the context of environmental health, public participation in waste management has historically suffered from a lack of immediate incentive or interactive feedback. Background of the Topic:

          The rapid expansion of urban populations and accelerated industrialization have led to an unprecedented increase in global solid waste generation. According to recent environmental studies, inadequate waste management is a primary contributor to soil degradation, water contamination, and escalating greenhouse gas emissions [20]. While stationary sorting facilities have seen advancements with integrated conveyor-belt AI systems, the collection processes at ground levels such as in public parks, university campuses, and municipal street remain predominantly reliant on manual labor. Recently, the intersection of autonomous robotics and edge-deployed deep learning has opened new avenues for distributed environmental maintenance. However, deploying these technologies in dynamic, unstructured public spaces presents challenges in both technical execution and public engagement.

Problem Definition:

           Current automated waste collection systems face two significant limitations. First, traditional cleaning robots function largely as blind vacuums or sweepers; they lack the visual intelligence required to distinguish between hazardous waste, recyclable materials, and non-target objects (e.g., wildlife or personal belongings), leading to inefficient and sometimes damaging collection cycles. Second, existing solutions are entirely utilitarian and fail to engage the public. They operate in isolation, missing an opportunity to educate communities or incentivize human participation in environmental cleanup efforts. There is a distinct lack of interactive frameworks that bridge the gap between autonomous environmental monitoring and human-driven gamified engagement.

 Need for the Project:

          To address the challenges, there is a critical need for an interactive, intelligent mobile platform capable of precise waste classification and active user engagement. A system is required that can operate remotely in real-world conditions, processing complex visual data locally without relying heavily on high-latency cloud computing. Furthermore, to combat the lack of public interest in waste management, the solution must incorporate a gamified element—transforming the mundane task of trash collection into an interactive experience or competition. Such a system requires a novel scoring mechanism that evaluates not just the act of collection, but the ecological value and physical volume of the extracted waste, thereby promoting targeted environmental impact.

 

Objective of the Work.

          The primary objective of this project is to conceptualize, design, and implement the Gamified Cleaning Bot (GCBOT)—a multi-tiered, teleoperated robotic system for intelligent waste collection. The specific goals of this work are to:

Develop an Edge-to-Fog Architecture: Establish a low-latency communication pipeline that bridges an Arduino-based hardware controller with an onboard Raspberry Pi and a high-compute Host machine over TCP sockets.

Implement Real-time Visual Intelligence: Integrate and fine-tune a lightweight YOLOv10 object detection model to accurately classify various waste categories in dynamic environments using a downscaled MJPEG video stream.

Design a Hybrid Gamification Engine: Create an automated scoring algorithm that evaluates collected waste utilizing a dual-parameter approach: optical classification (via YOLOv10 to determine base environmental impact) and physical mass sensing (via an integrated HX711 load cell).

Deploy a Responsive Control Interface: Build a Flask-based web application that provides operators with a seamless, mobile-optimized dashboard for remote execution, live telemetry visualization, and real-time score tracking.

Literature Review

 Previous Works and Existing Systems

The application of robotics and computer vision in waste management has been extensively explored in recent years, primarily driven by the need to automate hazardous and labor-intensive processes. Early iterations of automated waste collection focused on static smart bins. Authors such as [1] proposed IoT-enabled garbage receptacles that utilize ultrasonic sensors and microcontrollers to monitor fill levels and transmit data via GSM networks to municipal servers. While effective for route optimization, these systems lacked mobility and dynamic collection capabilities.

Advancements in mobile robotics led to the development of autonomous guided vehicles (AGVs) for street cleaning. Studies like the one conducted by [2] introduced vision-based path-finding robots capable of sweeping debris. However, these systems were largely "blind" to the nature of the waste they collected. The integration of Convolutional Neural Networks (CNNs) marked a significant turning point. Existing literature details stationary conveyor-belt sorting systems utilizing robust architectures like SSD or early YOLO variants (e.g., YOLOv3 and YOLOv4) to separate plastics, metals, and glass in recycling plants with high accuracy [3]. Recently, edge-deployed vision models on mobile platforms, such as Raspberry Pi and Jetson Nano, have been investigated for real-time trash detection, though often struggling with the trade-off between inference speed (framerate) and detection accuracy on low-power devices.

Furthermore, the concept of gamification in environmental initiatives has been tested primarily through mobile software applications rather than hardware. Platforms exist where users manually photograph trash to earn "green points" [4], effectively crowdsourcing the cleanup process without automating the physical collection.

Comparison

When comparing the proposed GCBOT with existing smart waste solutions, the divergence lies heavily in systemic integration and the scoring paradigm:

Stationary Sorters vs. Mobile Edge Control: 

Unlike industrial sorting facilities that rely on heavy, centralized server clusters, the GCBOT employs a multi-tiered edge-to-fog architecture. It pushes hardware actuation to an Arduino, network streaming to a Raspberry Pi, and heavy YOLOv10 inference to a localized host machine. This distribution ensures zero-latency mobility compared to cloud-reliant robots.

Volumetric vs. Optical Classification: 

Traditional automated bins classify waste purely by volume (using ultrasonic/IR sensors) or purely by optical category (using cameras). GCBOT introduces a unique hybrid assessment, synchronizing the optical classification from YOLOv10 with physical mass data from an HX711 load cell.

Software Gamification vs. Hardware Gamification:

Previous gamified cleanup efforts rely entirely on the user's honesty and manual labor via smartphone apps. The GCBOT turns the physical collection process itself into a teleoperated arcade-style experience, marrying the software reward system directly with the physical action of the robotic arm.

Limitations of Existing Methods

Despite the progress in technology, several critical limitations persist in the current technological landscape:

Incomplete Evaluation Metrics: Current vision-only sorting robots cannot determine the physical mass of an object. A large cardboard box might be optically classified, but without weight sensors, the system cannot accurately quantify the amount of material recovered. This leads to inaccurate automated auditing.

High Latency in Mobile Teleoperation: Remote-controlled utility robots often suffer from control lag when video processing and network communication share the same thread on a single embedded device. This makes real-time, precise teleoperation frustrating and impractical.

Lack of Public Engagement Mechanisms: Current automated sweepers and sorters are strictly utilitarian. They operate passively around humans and fail to foster any behavioural change or ecological awareness in the public.

Model Overhead: Utilizing older, heavier CNNs on autonomous robots often necessitates expensive onboard GPUs or high-latency cloud computing, drastically increasing the physical weight, battery consumption, and cost of the mobile unit.

The GCBOT is explicitly designed to address these limitations by implementing a non-blocking TCP network protocol for low latency, utilizing the computationally efficient YOLOv10 architecture, and pioneering a hybrid AI-Weight gamification algorithm to engage human operators.

Methodology / System Design

Overall System Architecture

The foundation of the GCBOT is a highly distributed three-tier architecture (Edge-to-Fog) designed to decouple heavy AI processing from physical hardware control.

Fog/Host Node: A high-computation central server hosting a Python Flask backend. This layer is strictly responsible for running the YOLOv10 deep learning model, calculating gamification scores, and rendering the user-facing web dashboard.

Edge Node (Network Bridge): A Raspberry Pi 4 acts as the intermediary. It handles local camera streaming (MJPEG) and converts incoming TCP network packets from the host into direct serial communication for the microcontroller. Furthermore, applying gamification—the integration of game mechanics into non-game contexts—presents a unique opportunity to incentivize public participation in environmental cleanup [16],[17].

Hardware Controller Layer: An Arduino Uno handles the lowest level of physical execution. It directly drives the motor systems and continuously monitors analog sensors without being interrupted by complex network or AI tasks.

Components Used:

The robotic platform integrates standard off-the-shelf components configured for maximum reliability:

Locomotion: A standard 4-wheel drive (4WD) chassis utilizing four geared DC motors, regulated by an L298N dual H-bridge motor driver [18].

Manipulation: A custom-built robotic arm powered by four independent servo motors (managing both vertical lift and gripper actuation) [19], stepped down by a dedicated 5V buck converter.

Processing Units: An Arduino Uno for real-time actuation, paired with a Raspberry Pi 4 (Model B) for local networking and video acquisition.

Sensory Modules: A 1kg aluminium load cell equipped with an HX711 amplifier module for precise mass calculation [11], alongside a standard USB webcam for visual feed generation.

Working Principle:

The operational workflow of the GCBOT relies on minimizing the delay between user command, AI assessment, and physical actuation:

Perception and Streaming: The onboard camera captures raw frames and broadcasts them as an aggressive MJPEG stream. The Host machine intercepts this stream, dynamically downscales frames to 416 × 416 pixels, and evaluates them through the custom-trained YOLOv10 model. Detected objects are bounded, classified (e.g., "Plastic Bottle", "Broken Glass"), and their physical coordinates are overlayed onto the web interface.

Actuation: The operator, observing the annotated stream via a smartphone browser, utilizes HTML5 pointer events to drive the robot towards the target. Pressing a directional button sends an HTTP request, which the Flask server instantly pipes over a raw socket (with TCP_NODELAY enabled) to the Pi. The Pi flashes a byte-level serial command to the Arduino, which manipulates the L298N pins.

The Scoring Event: Upon successful collection, the operator presses the "DROP OBJECT" button. This triggers a synchronized macro sequence: the arm servos release the object; the system non-blockingly waits for physical stabilization; the Arduino polls the HX711 module to deduce the delta weight; and the Host machine computes the final gamified score using the formula Score = Base × (1 + Weight/100), utilizing the last visually detected class.

Block Diagram Explanation:

The system block diagram illustrates the hierarchical, top-to-bottom flow of data and control across the three architectural tiers.

Top Block (Tier 3: Host / Fog Layer): At the top of the hierarchy sits the Host Layer. The user interacts with the system via the Web Browser, which communicates bidirectionally over HTTP and WebSocket with the Flask Web Server. The Flask Server then passes video frames down to the YOLOv10 Inference engine to extract object classes. Notably, there is a prominent upward path originating from the Edge layer, showing the continuous "MJPEG Video Stream" feeding directly into the Host environment.

Middle Block (Tier 2: Edge Layer): Acting as the central network bridge, the Edge Layer receives "Raw Video" from the USB Web Camera and funnels it into the Raspberry Pi. From the Host layer above, "TCP Control Commands" flow downward into the Pi, which processes and translates them for hardware execution.

Bottom Block (Tier 1: Hardware Layer): A bidirectional "Serial USB" tie connects the Raspberry Pi above to the Arduino microcontroller below. The Arduino serves as the distribution hub for the hardware layer. It sends "PWM Signals" to actuate both the custom arm assembly (represented by the 4x Servo Motors) and the L298N Motor Driver, which subsequently supplies power to the 4x DC Motors traversing the ground. Simultaneously, mechanical force is captured by the 1kg Load Cell, passed as an "Analog Signal" through the HX711 amplifier, and delivered back to the Arduino as "Digital Weight Data".

This vertical visualization distinctly highlights how heavy computational tasks and user interfaces remain isolated at the top (Host), while real-time sensory polling and hardware actuation are pushed to the bottom (Microcontroller), bridged exclusively by the local networking capabilities of the Edge Node in the center.

 Implementation

Hardware setup:

The physical construction of the GCBOT involves the integration of motive, actuation, and sensory subsystems onto the 4WD chassis.

Motive Drive: The L298N driver is wired to the Arduino Uno, taking 12V input from a LiPo battery and distributing it to the four DC motors.

Robotic Arm: The four servo motors are powered via a dedicated 5V buck converter (to prevent brownouts on the microcontroller) and are controlled via PWM pins on the Arduino.

Sensing and Computing: The HX711 module links the load cell to the Arduino's analog pins. The Arduino is connected via a short USB-B to USB-A cable to the onboard Raspberry Pi 4. The wide-angle webcam is mounted above the arm assembly, plugged directly into the Pi’s USB 3.0 port.

Software Used:

The software stack spans multiple environments to cater to the edge-to-fog architecture:

Microcontroller: Written in C++ using the Arduino IDE.

Edge Node (Raspberry Pi): Python 3.9+ utilizing OpenCV [15] (for grabbing and streaming MJPEG frames) and custom socket scripts for TCP networking.

Host Machine: Python 3.x environment. The web backend leverages Flask [14] for HTTP routing, WebSocket for real-time score updates, and the Ultralytics library [5] to run the custom-trained YOLOv10 object detection model [13].

Frontend Interface: Pure HTML, CSS, and vanilla JavaScript to ensure maximum cross-browser mobile compatibility without the overhead of heavy frameworks like React.

Code logic:

The implementation logic is driven by asynchronous and non-blocking philosophies:

The Video Pipeline: app.py on the host machine maintains a background daemon thread that continuously pulls bytes from the Pi's MJPEG stream (Port 9999). It constructs frames, passes them through model(frame) for YOLOv10 inference, draws bounding boxes, and yields the annotated frame via a Flask /video feed route.

The Control Loop: When a user holds a directional button on the web UI, an HTTP GET request is triggered. app.py intercepts this and immediately pushes an ASCII command (e.g., "F" for forward) over a persistent TCP socket to the Pi [10]. The Pi's pi_control.py script reads this socket and writes the byte to /dev/ttyACM0 (the Arduino).

The Scoring Logic: The "Drop" command initiates the hybrid assessment. The Pi instructs the Arduino to open the gripper. The Arduino waits non-blocking for 2 seconds for the load cell to stabilize, reads the HX711, and writes the string "W:(grams)" back to the Pi. The Pi forwards this to the Host. The Host matches this weight with the "last detected class" saved in the video thread memory, calculates the Base * (1 + Weight/100), and emits a WebSocket event to update the user's score on the screen dynamically.

Results and Discussion

Output of the System:

The GCBOT system was successfully prototyped and deployed in a controlled test environment simulating a public space with scattered waste. The Flask-based web interface proved highly reliable across both desktop and mobile browsers, successfully rendering the live MJPEG stream annotated with bounding boxes. Upon classification of an object, the UI accurately displayed the detected class (e.g., "Plastic Bottle"), and upon collection, the system seamlessly executed the customized Base * (1 + Weight/100) algorithm, instantly updating the gamified scoreboard.

 Observations:

During operational testing, several key observations were made regarding the hybrid scoring methodology. The physical integration of the HX711 load cell under the collection receptacle provided highly consistent data, though a mandatory non-blocking delay of 2 seconds was required after dropping an object to allow mechanical vibrations to settle before sampling the weight. Visually, the YOLOv10 model handled overlapping objects reasonably well. However, in low-light conditions, the detection confidence occasionally dropped below the 0.25 threshold. In these instances, the system successfully defaulted to the volumetric fallback algorithm (Score = max (1, Weight/10)), proving the robust redundancy of the hybrid telemetry approach.

 Performance Analysis:

The performance of the system was evaluated based on AI inference speed, network latency, and physical response times:

Object Detection Accuracy: The custom-trained YOLOv10 model achieved a mean Average Precision (mAP) of roughly [50 - 70] % across the 18 trained waste classes. Downscaling the camera feed to 416x416 prior to inference yielded a consistent [30 - 40 FPS] on the host machine.

Network Command Latency: By implementing pure TCP sockets and explicitly disabling Nagle's algorithm (TCP_NODELAY), the control lag between pressing a button on the UI and the physical rotation of the DC motors was reduced to near-zero (averaging [< 10ms]). This allowed for smooth, arcade-style teleoperation.

Weight Accuracy: The load cell demonstrated an accuracy margin of [± 2 grams], which was more than sufficient for the scoring algorithm's granularity.

 Conclusion

 Summary of Work:

This paper presented the design and implementation of the Gamified Cleaning Bot (GCBOT), a novel approach to addressing localized solid waste management through the integration of edge robotics, deep learning, and user gamification. A custom 4-wheel drive robotic platform equipped with a servo-actuated arm was constructed and networked via a three-tiered architecture encompassing an Arduino, a Raspberry Pi, and a high-performance Fog/Host server.

Achievements:

The core achievement of this project is the successful deployment of a dual-parameter "Hybrid AI-Weight" scoring algorithm. By synchronizing the optical intelligence of a fine-tuned YOLOv10 classification model with the physical mass data extracted from an onboard load cell, the GCBOT accurately assigns a gamified ecological value to collected waste. Furthermore, by isolating heavy AI processing on the host machine and utilizing TCP_NODELAY network protocols, the system achieved the low-latency responsiveness required for real-time mobile teleoperation.

Final Outcome:

Ultimately, GCBOT transforms the mundane and hazardous chore of manual garbage collection into an engaging, interactive, and competitive digital experience. The system provides a robust, scalable foundation for incentivizing public participation in environmental cleanup efforts.

Future Scope

 Possible Improvements:

While the current iteration of the GCBOT is highly functional, several improvements can be made. Future models will focus on migrating the YOLOv10 inference directly to the edge using dedicated neural processing units (such as the Google Coral TPU or NVIDIA Jetson Nano), which would eliminate the need for a localized host computer and allow the robot to operate entirely autonomously over long-range cellular networks (4G/5G). Additionally, integrating ultrasonic collision-avoidance sensors would prevent accidental impacts caused by operator error during remote control.

Extensions:

The gamification engine can be significantly expanded. Future extensions of this work could involve linking the Flask backend to a cloud database (like Firebase or AWS) to create global leaderboards, allowing users operating different GCBOTs in various cities to compete in global cleanup events over the internet. Furthermore, the object detection model could be continuously trained via federated learning, allowing the robot fleet to passively learn to identify locally specific types of trash over time.
